# Phase 121 — Virtual Pi (WS3): guided hookup, print view, pi_client config export

## Why

Phases 119–120 make the UI the source of truth for wiring. But the operator still
hand-copies that truth twice: once onto the physical wires, once into
`pi_client/config.yaml`. Both copies drift. This phase generates both artifacts
from the same data the board renders.

## Gaps found while scoping

1. **config.yaml is hand-maintained.** `PiSetupGuide.vue` shows a static snippet
   ("config.yaml snippet (Phase 51 wiring)") and `pi_client/setup.sh` copies an
   example file. The DB already knows every sensor's driver, pin, device, and
   interval — rendering the file is pure serialization.
2. **No drift detection.** If someone rewires in the UI but never updates the Pi
   (or vice versa), nothing notices. Phase 114 (Pi edge integrity) added
   device-key auth; confirm whether the heartbeat carries config identity — if
   not, add a config hash to the heartbeat payload.
3. **Driver hookup knowledge lives only in people's heads.** The taxonomy
   (Phase 90) knows drivers exist but not their physical pin requirements
   (VCC voltage, data pin count, bus type). Small data addition, big usability
   win — it's what makes the virtual Pi genuinely "show how to wire up the Pi".

## Design notes

- **WS1:** hookup data belongs in the taxonomy payload (server-owned, one place),
  mirrored in `deviceTaxonomy.fallback.js`. Shape:
  `hookup: [{ wire: 'VCC', to: 'any 3v3 pin', role: 'power3v3' }, { wire: 'DATA', to: 'chosen GPIO', role: 'gpio' }, ...]`.
- **WS3:** Go-side rendering (template or yaml.Marshal of a struct) so the API is
  the single generator; UI just downloads. Endpoint requires farm-member auth on
  the owning farm. Include a `# generated by gr33n — do not hand-edit` header
  with timestamp + device UID.
- **WS4:** cheapest sound design — Pi includes `config_sha256` in its existing
  heartbeat; API stores it on the device row and compares lazily when the board
  loads. No push mechanism needed this phase.
- Print view: a `?print=1` mode of the existing view + `@media print` CSS beats
  a separate route or PDF generation.

## Out of scope

- Auto-pushing config to the Pi (pending_command channel could carry it later)
- Fritzing-style breadboard illustrations — step lists + pin highlights only
- Multi-Pi orchestration views

## Acceptance

- [ ] Selecting DHT22 on a pin shows its 3-wire hookup steps with the right physical pins highlighted
- [ ] Print dialog output is legible on one–two A4/letter pages for the demo farm
- [ ] Downloaded config.yaml validates against pi_client's loader (run `gr33n_client.py --validate-config` if present, else add it)
- [ ] Changing a pin in the UI flips the device to "stale wiring" after next heartbeat with the old hash
- [ ] openapi.yaml documents `/devices/{id}/pi-config`; `make audit-openapi` passes
- [ ] Pi setup guide links to export; INSTALL.md updated
